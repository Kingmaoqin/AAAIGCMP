## 2. 当前系统已经具备的核心组件

### 2.1 总装配入口

文件：
- [bootstrap.py](/home/xqin5/agenticAI/agent_core/bootstrap.py)

这个文件负责把整个系统装起来。`build_default_runner()` 会完成这些事情：

- 读取 `LOCAL_LLM_ENDPOINT` 或 `OPENAI_COMPAT_BASE_URL`
- 如果配置了 LLM 端点，就实例化 `OpenAICompatibleLLMClient`
- 创建 `SkillRegistry`
- 注册内置 skills
- 创建 `Planner`
- 创建 `SkillRouter`
- 创建 `SchemaVerifier`、`EvidenceVerifier`
- 创建 `Finalizer`
- 最后返回一个 `StateMachineRunner`

也就是说，当前系统真正的统一入口不是某个 Web 服务，而是 `build_default_runner()`。

### 2.2 统一数据结构

文件：
- [schemas.py](/home/xqin5/agenticAI/agent_core/schemas.py)

当前系统的所有中间结果都围绕几种 dataclass 流动：

- `TaskSpec`
  任务解析后的结构化定义
- `PlanStep`
  执行计划中的单个步骤
- `Artifact`
  每个 skill 产出的中间结果
- `TraceEvent`
  运行过程中的轨迹事件
- `FinalResponse`
  最终输出
- `RunState`
  一次运行的总状态容器

这层的意义是把“任务、步骤、产物、轨迹、最终结果”统一建模。后续 parser、planner、router、skills、verifier 都围绕这些对象协作。

### 2.3 任务解析器

文件：
- [parser.py](/home/xqin5/agenticAI/agent_core/core/parser.py)

`TaskParser.parse()` 负责把原始 query 转成 `TaskSpec`。当前逻辑是规则驱动，不是 LLM 驱动。

它现在主要做三类判断：

- 是否包含数学表达式
- 是否要求依据文档/handbook/policy/evidence 回答
- 由此确定任务 family

当前支持的 family：

- `general_qa`
- `calculation_only`
- `mixed_analysis`

同时它还会写入：

- `constraints`
- `output_schema`
- `eval_focus`
- `budget`

因此 parser 不是只做字符串清洗，而是在给后续 planner 和 router 提供任务约束。

### 2.4 策略层 Policy

文件：
- [builtin.py](/home/xqin5/agenticAI/agent_core/policies/builtin.py)

Policy 的作用是给不同任务 family 提供默认执行骨架。

当前有三种：

- `GeneralQAPolicy`
- `CalculationPolicy`
- `MixedAnalysisPolicy`

它们的共同特点是：

- 直接返回一个 `list[PlanStep]`
- 每个 step 预先定义好：
  - `goal`
  - `inputs_needed`
  - `candidate_skills`
  - `expected_artifact`

所以 policy 不是最终执行逻辑，而是“默认计划模板”。当 LLM planner 不可用或失败时，系统会退回到这里。

### 2.5 Planner

文件：
- [planner.py](/home/xqin5/agenticAI/agent_core/core/planner.py)

Planner 的入口是 `Planner.build_plan()`。

当前工作机制：

1. 先用对应 policy 生成 fallback plan
2. 如果没有 LLM 或 LLM 不健康，直接返回 fallback plan
3. 如果有 LLM，尝试用 LLM 生成 plan
4. 如果 LLM 失败，返回 `rule_fallback`

当前 LLM planning 有两级逻辑：

- 一级是简化序列规划 `_build_with_simple_llm_sequence()`
  - 只让模型输出 `parsed_query,retrieval_result,calculation_result,report` 这种 artifact 序列
  - 再把序列还原成 `PlanStep`
- 二级是完整 JSON 规划 `_request_plan_payload()`
  - 让模型输出严格 JSON 的 steps
  - 如果模型输出脏文本，再尝试 JSON 抽取
  - 如果还是失败，再从自然语言里恢复步骤

这部分是之前让本地 LLM 真正跑进 planning 的关键修复点。

### 2.6 Router

文件：
- [router.py](/home/xqin5/agenticAI/agent_core/core/router.py)

Router 的入口是 `SkillRouter.select()`。

它的执行过程是：

1. 根据 `step.candidate_skills` 从 registry 里取出候选 skill
2. 用 `can_handle()` 过滤不适用 skill
3. 如果只有一个 skill，直接选它
4. 如果有多个 skill 且 LLM 可用，则尝试 LLM 路由
5. 如果 LLM 路由失败，再回退到规则选择

当前 LLM 路由也有两级：

- `_select_with_simple_prompt()`
  - 直接让模型只返回 skill 名称
- `_select_with_llm()`
  - 让模型返回 JSON
  - 如果 JSON 不稳定，就从自然语言中恢复 skill 名称

此外还有 `_prefer_llm_skill()`，当某一步有 `llm_*` 技能时，它会优先保留 LLM-native 技能。

### 2.7 Skill Registry 与 Skills

文件：
- [registry.py](/home/xqin5/agenticAI/agent_core/skills/registry.py)
- [base.py](/home/xqin5/agenticAI/agent_core/skills/base.py)
- [builtin.py](/home/xqin5/agenticAI/agent_core/skills/builtin.py)

Registry 的作用是注册和检索技能。

当前内置的技能有：

- `parser`
  规则版 query parser
- `llm_query_parser`
  LLM 版 query parser
- `retrieval`
  基于内置 mock 文档做关键词检索
- `calculator`
  规则算术计算
- `report_writer`
  模板化报告生成
- `llm_report_writer`
  LLM 报告生成

这里要特别说明当前 retrieval 的实际状态：

- 现在不是真正的向量 RAG
- 也不是外部知识库检索
- 它只是对 [builtin.py](/home/xqin5/agenticAI/agent_core/skills/builtin.py) 里的 `MOCK_DOCUMENTS` 做关键词重叠打分

当前 mock 文档包括：

- `Office Expense Handbook`
- `Travel Policy`
- `Team Operations Guide`

所以目前系统的“知识来源”仍然是内置字典，不是外部文档系统。

### 2.8 LLM 客户端

文件：
- [llm_client.py](/home/xqin5/agenticAI/agent_core/services/llm_client.py)

`OpenAICompatibleLLMClient` 是当前系统对外部 LLM 的统一适配层。

它现在支持：

- 本地 `LOCAL_LLM_ENDPOINT`
- 未来其他人的 OpenAI-compatible API
- 可选的 `LLM_API_KEY`
- 可选的 `LLM_MODEL`

当前它提供的核心能力：

- `is_healthy()`
- `chat_completion()`
- `chat_json()`
- `extract_json_object()`
- `extract_skill_name()`
- `coerce_report_content()`

这里很关键的一点是：现在系统没有把外部 LLM API 直接散写在 planner/router/skills 里，而是先收口到这个 client 层。

### 2.9 Verifier 与 Finalizer

文件：
- [verifier.py](/home/xqin5/agenticAI/agent_core/core/verifier.py)
- [finalizer.py](/home/xqin5/agenticAI/agent_core/core/finalizer.py)

`SchemaVerifier` 当前检查：

- `answer` 是否存在且为字符串
- `summary` 是否存在且为字符串
- `evidence_ids` 是否存在且为 `list[str]`

`EvidenceVerifier` 当前检查：

- 如果任务要求 evidence，则 `evidence_ids` 不能为空
- `evidence_ids` 中的每个 artifact id 必须在 `run_state.artifacts` 中存在

`Finalizer.finalize()` 则负责把 report artifact 转成最终 `FinalResponse`。

### 2.10 State Machine

文件：
- [state_machine.py](/home/xqin5/agenticAI/agent_core/core/state_machine.py)

这是真正的主执行引擎。

`StateMachineRunner.run()` 的顺序非常清楚：

1. `PARSE_TASK`
2. `CLASSIFY_TASK`
3. `BUILD_PLAN`
4. 顺序执行每个 `PlanStep`
5. `FINAL_VERIFY`
6. `FORMAT_OUTPUT`
7. `DONE`

在 step 循环内部，每一步都按相同模式运行：

1. `router.select()`
2. `decision.skill.run()`
3. 把产物写入 `run_state.artifacts`
4. 记录 trace

最终它还会把运行结果持久化到：

- `runs/run-xxx_state.json`
- `runs/run-xxx_trace.json`

## 3. 当前系统的真实执行链路

现在整个系统的运行链路可以概括成：

`query -> TaskParser -> Policy -> Planner -> Router -> Skill -> Artifact -> Verifier -> Finalizer -> RunState/Trace`

把它展开成代码上的实际过程就是：

1. 用户输入 query
2. `TaskParser.parse()` 生成 `TaskSpec`
3. 根据 `TaskSpec.family` 选择 policy
4. `Planner.build_plan()` 生成 plan
5. 针对每个 step：
   - `SkillRouter.select()` 选 skill
   - `skill.run()` 产出 artifact
   - `run_state.add_artifact()` 保存产物
   - `_append_trace()` 记录事件
6. 找到最终 `report` artifact
7. `SchemaVerifier` 和 `EvidenceVerifier` 校验
8. `Finalizer.finalize()` 生成 `FinalResponse`
9. `RunState` 和 `trace` 落盘

这个结构的优点是调试清晰。因为每一步的输入、输出、所选 skill、路由模式、错误信息都能在 state/trace 里看到。

## 7. 对当前系统问题的逐条代码级分析

下面逐条回应你提出的几个问题。

### 7.1 “框架上代码抽象的东西太多，其核心就是围绕着这个函数在包装”

结论：
- 这句话部分属实，而且是当前系统最明显的可用性问题之一。

原因：

从代码结构看，真正控制一次任务执行的核心确实主要集中在：

- [state_machine.py](/home/xqin5/agenticAI/agent_core/core/state_machine.py)
  的 `StateMachineRunner.run()`

其他模块大多是在为这条主执行链提供前后置能力：

- `TaskParser.parse()`
- `Planner.build_plan()`
- `SkillRouter.select()`
- `skill.run()`
- `SchemaVerifier.verify()`
- `EvidenceVerifier.verify()`
- `Finalizer.finalize()`

所以从“控制流”角度讲，当前系统的确是围绕 `run()` 在做包装。

但这并不完全等于“抽象是无意义的”。这些抽象目前至少承担了三个作用：

- 把 planning、routing、skill execution 分层
- 把 LLM 适配从业务逻辑里分离出去
- 让每一步都有独立替换点

真正的问题不在于“有抽象”，而在于：

- 当前目录分层已经开始增加跳转成本
- 对新读代码的人来说，理解入口要在多个文件之间来回切
- 抽象层目前还没有配套一个非常直接的“最小扩展路径”

代码证据：

- 主控制流集中在 [state_machine.py](/home/xqin5/agenticAI/agent_core/core/state_machine.py)
- 总装配集中在 [bootstrap.py](/home/xqin5/agenticAI/agent_core/bootstrap.py)
- 抽象点分散在 `parser/planner/router/skills/verifier/finalizer`

因此，这个批评是成立的，但更准确的表述应当是：

> 当前系统已经形成了可扩展抽象层，但对开发者体验而言，抽象层偏多，入口说明和扩展路径说明还不够直接。

### 7.2 “数据融合之类的 RAG，有点是在顺序执行多个 LLM”

结论：
- 这句话对“当前代码现状”并不完全成立，但对“未来扩展风险”是成立的。

为什么说当前不完全成立：

当前仓库里并没有真正的 RAG。

当前 retrieval 是：

- [builtin.py](/home/xqin5/agenticAI/agent_core/skills/builtin.py)
  里的 `RetrievalSkill`

它只是：

- 从 `parsed_query` 取关键词
- 和 `MOCK_DOCUMENTS` 做词项重叠
- 选分数最高的文档

所以现在并没有出现：

- 检索前一个 LLM
- 重写 query 再一个 LLM
- rerank 再一个 LLM
- synthesis 再一个 LLM

这种真正意义上的多段 LLM RAG pipeline。

但为什么说这个担忧是对的：

如果后续按最直观方式扩展，很容易变成：

- query rewrite 用一个 LLM
- retriever 前后再插一个 LLM
- reranker 再用一个 LLM
- 最终 answer synthesis 再用一个 LLM

这样就会形成串行多 LLM 调用，延迟和失败率都会上升。

所以这条批评针对“当前代码”并不准确，针对“后续容易演化的方向”是准确的。

### 7.3 “新用户得先把程序跑通，再一个一个对着抽象接口去看逻辑，跳来跳去，才能开始写代码”

结论：
- 这条基本属实。

原因：

新用户要扩展一个技能，通常得先理解这几个位置：

- [bootstrap.py](/home/xqin5/agenticAI/agent_core/bootstrap.py)
  看系统如何装配
- [base.py](/home/xqin5/agenticAI/agent_core/skills/base.py)
  看 skill 接口
- [registry.py](/home/xqin5/agenticAI/agent_core/skills/registry.py)
  看 skill 如何注册
- [builtin.py](/home/xqin5/agenticAI/agent_core/skills/builtin.py)
  看已有 skill 的写法
- [builtin.py](/home/xqin5/agenticAI/agent_core/policies/builtin.py)
  看 step 是怎么声明 candidate skills 的
- [router.py](/home/xqin5/agenticAI/agent_core/core/router.py)
  看 skill 最终怎么被选中

所以新用户如果要“加一个新能力”，不是只改一个文件。

这就导致两个现实问题：

- 理解路径长
- 第一处改动点不够明显

因此，这个问题是真实存在的，属于“扩展体验差于运行体验”。

### 7.4 “对多 LLM 合作的情况，如果其中一个 LLM 调用异常，可能整个任务就报错”

结论：
- 这句话部分属实，但不能一概而论。

为什么说不完全属实：

当前代码里，很多 LLM 调用已经做了降级保护。

例子：

- `Planner.build_plan()` 中，LLM planning 失败会回退到 fallback plan
- `SkillRouter.select()` 中，LLM routing 失败会退回规则选择
- `LLMQueryParserSkill.run()` 中，LLM parsing 失败会退回规则 parse
- `LLMReportWriterSkill.run()` 中，LLM report 失败会退回规则 report

这些都说明：

- 当前系统不是“任意一个 LLM 出错，整条链就死”
- 已经做了局部 fallback

为什么又说部分属实：

当前系统仍然存在几类会导致整条任务失败的情况：

- 某一步只有一个技能，且该技能本身报错
  - 例如 retrieval 没拿到 `parsed_query`
  - 或 calculator 没拿到表达式
- 最终 verifier 不通过
  - `StateMachineRunner.run()` 会直接抛出异常
- 非 LLM 的确定性步骤没有替代路径
- 没有 retry、熔断、step 级错误隔离

代码证据：

- fallback 存在于 [planner.py](/home/xqin5/agenticAI/agent_core/core/planner.py)
- fallback 存在于 [router.py](/home/xqin5/agenticAI/agent_core/core/router.py)
- fallback 存在于 [builtin.py](/home/xqin5/agenticAI/agent_core/skills/builtin.py)
- 全局失败出口在 [state_machine.py](/home/xqin5/agenticAI/agent_core/core/state_machine.py)

因此，更准确的结论是：

> 当前系统对单点 LLM 失败已经有一定韧性，但对多步骤任务的整体容错还不够强，仍然可能因关键步骤失败而整条任务中断。

### 7.5 “好像整个框架单任务阻塞执行”

结论：
- 这条是成立的。

原因非常直接：

- [state_machine.py](/home/xqin5/agenticAI/agent_core/core/state_machine.py)
  中 `run()` 是同步函数
- 它内部用 `for step in run_state.plan:` 顺序执行所有 step
- 没有 `async`
- 没有线程池
- 没有并行执行多个 skill
- 没有任务队列
- 没有调度器

因此当前系统是：

- 单次 run 内部串行执行
- 一个 step 完成后才到下一个 step
- 某个 LLM 调用慢，会直接阻塞当前 run

从工程上讲，这不是 bug，而是 MVP 阶段的简化设计。

但如果未来要做：

- 多文档检索
- 多模型协作
- 多用户服务
- benchmark 批量运行

那这个串行执行模型会很快成为瓶颈。

## 8. 当前系统的真正优点与真正短板

### 8.1 当前真正的优点

- 已经形成了一个能运行的 agent orchestration core
- 数据结构统一
- trace 清晰
- LLM 接口已做兼容层
- 规则与 LLM 双模式可切换
- 出现脏 JSON 时已有一定恢复能力
- 可以从本地 LLM 平滑迁移到其他 OpenAI-compatible API

### 8.2 当前真正的短板

- retrieval 还是 mock，不是真实 RAG
- 扩展路径不够短
- step 执行是同步串行
- 还没有 session / memory / handoff
- 错误恢复主要靠 fallback，不是完整容错框架
- 目前更像单次任务 runner，不是完整服务化系统

## 9. 对后续改进的建议

如果要让这个仓库更适合作为团队共同开发的核心，我建议优先做下面几件事。

### 9.1 降低扩展门槛

建议补一个“新增 skill 的最短路径”文档和模板，至少明确：

- 新 skill 写在哪
- 需要实现哪些方法
- 如何注册
- policy 里怎么接进去
- 怎样跑最小测试

### 9.2 把 mock retrieval 升级成真实 RAG

建议优先替换 `RetrievalSkill`：

- 支持本地文档目录
- 支持 chunking
- 支持 embedding / vector store
- 支持 evidence 片段级引用

### 9.3 增强 step 级容错

建议增加：

- retry
- timeout
- step-level fallback policy
- 可选跳过非关键步骤
- richer failure artifact

### 9.4 解决串行阻塞

后续可以考虑：

- async runner
- 并发执行独立 step
- batch eval runner
- service queue

## 10. 总结

当前 `/home/xqin5/agenticAI` 已经不是一个概念性框架，而是一个真实可运行的 Agentic AI 核心原型。它已经具备：

- 任务解析
- 任务策略
- 计划生成
- 技能路由
- 技能执行
- 结果校验
- 最终汇总
- 轨迹记录

并且已经通过了一个真实的小任务验证，证明本地 LLM 已经可以真正参与 planning、routing、query parsing 和 report writing。

但你的几个批评里，有三点是确实成立的：

- 抽象层偏多，开发者理解成本偏高
- 新用户扩展门槛偏高
- 当前执行是单任务串行阻塞

另外两点则需要更精确地表述：

- “RAG 是顺序执行多个 LLM” 这不是当前事实，但确实是未来扩展风险
- “一个 LLM 调用异常就整个任务报错” 当前并非总是如此，但全链路容错还不够强

所以现在这个系统最准确的定位是：

> 它已经是一个能跑通的小型 agent orchestration core，但还没有进化成易扩展、强容错、可服务化的大型 agent 平台。

这个判断是基于当前代码结构、真实运行状态和测试结果得出的。
