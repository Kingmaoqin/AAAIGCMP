# RL1128 vs RL0305 系统升级、算法修复与部署评估

## 1. 比较对象与结论

本文比较两个版本：

- 旧版系统：`/home/xqin5/RLDTALL/RL1128`
- 新版系统：`/home/xqin5/RLDTALL/RL0305`

结论先给清楚：

1. `RL0305` 相比 `RL1128` 已经不是“小修小补”，而是把数据构造、动力学模型、offline policy、online BCQ 路由、active learning/front-end、数据质量自适应这几条主链路都重构了一遍。
2. 在同域、同语义、质量较好的真实治疗时序数据上，`RL0305` 已经明显优于 `RL1128`，而且稳定性更好。
3. 在质量下降但仍然真实的退化数据上，`RL0305` 的鲁棒性明显提升，尤其是最坏种子和方差显著改善，但平均收益仍未稳定达到“放心部署”的水平。
4. 因此，当前系统可以说“对当前这类 ovarian longitudinal treatment 数据，已经接近可部署”，但**还不能说“任意 treatment 相关数据进来都能稳定维护成 digital twin 并给出高质量推荐”**。

## 2. 系统在做什么

系统的目标是：给定病人的治疗轨迹数据，维护一个可滚动更新的数字孪生状态，并据此输出治疗推荐。

核心分成四层：

1. 状态构造：把原始表格变量映射成时序状态 `s_t` 和动作 `a_t`。
2. 动力学模型：学习状态转移 `s_t -> s_{t+1}`。
3. 奖励/结局与 Q 模型：评估某状态下某动作的长期价值。
4. Online 更新：真实流式数据进入后，做小步更新和策略修正。

符号约定：

- `s_t`：第 `t` 个时间点的病人状态向量。
- `a_t`：第 `t` 个时间点的治疗动作。
- `r_t`：即时奖励，代表短期健康改善或缓解信号。
- `d_t`：终止标记，若轨迹结束则 `d_t = 1`。
- `gamma`：折扣因子，控制长期回报权重。
- `theta`：动力学模型参数。
- `phi`：奖励/结局模型参数。
- `psi`：Q 网络参数。
- `Q_psi(s,a)`：状态 `s` 下执行动作 `a` 的长期价值估计。

## 3. 旧版 RL1128 的主要问题

### 3.1 数据构造错误直接污染了动力学学习

旧版在 `run_on_ovdata_multilabel.py` 中对 `clinical_stage` 的编码存在两个问题：

1. 阶段字符串匹配顺序错误。
2. 编码值直接用 `1,2,3,4`，而后续状态和模型假设大多在 `[0,1]` 内。

旧版代码片段：

```python
if "stage i" in xl:
    return 1.0
if "stage ii" in xl:
    return 2.0
if "stage iii" in xl:
    return 3.0
if "stage iv" in xl:
    return 4.0
```

因为 `"stage ii"`、`"stage iii"`、`"stage iv"` 都包含子串 `"stage i"`，所以很多高分期被错误映射到 stage I。等于把一个重要临床静态特征压坏了。

此外，后续版本中还一度把 `state[9]` 塞成了“归一化 visit index”。这一维依赖整条轨迹长度，不满足单步状态转移假设，会让模型学习到不该学的时间泄漏。

### 3.2 Offline policy 的训练验证口径不一致

旧版的关键问题不是 CQL 公式错了，而是训练、验证、评估不是同一问题：

- 数据切分一开始偏向 transition 级随机切分，容易把同一轨迹同时泄漏到 train/val。
- validation Bellman target 与训练口径不完全一致。
- 训练内选模曾错误启用 safety cut，导致 best checkpoint 不是按真实 deployment 目标选出来的。

这会造成一个典型假象：训练日志看起来在收敛，但 holdout return 和真实 online 收益不稳定。

### 3.3 Online 路由和实现耦合严重

旧版存在几类工程问题：

- `ours/bcq` 在线上有路径会被错映射成 `cql`。
- BCQ policy 路径硬编码为 `./output/models/best_bcq_policy.d3`，导致换目录后容易静默回退。
- online batch 较窄时，BCQ updater 会把 `action_size` 重建成局部动作子集，直接破坏动作空间。
- front-end 的 `query_batch` 曾出现“记账成已查询，但没真的逐条送到 expert”的问题。

这些问题叠加后，旧系统很容易出现“指标看起来在跑，实际上算法没按预期工作”的假结果。

## 4. 新版 RL0305 做了哪些关键修改

## 4.1 状态构造与数字孪生底座修复

### 4.1.1 修复 `clinical_stage` 编码

新版把 stage 映射改成了临床顺序正确、数值范围也正确的归一化编码：

- stage I -> `0.25`
- stage II -> `0.50`
- stage III -> `0.75`
- stage IV -> `1.00`

算法含义：这一步不是“调参”，而是把状态空间定义重新修正到可学习区间内。否则模型输出经 `clamp(0,1)` 后，本来就不可能正确拟合 `2,3,4` 这种标签。

### 4.1.2 去掉违反转移假设的时间泄漏状态

把 `state[9]` 从“归一化 visit index”恢复为稳定质量位/常数位，避免模型学到“轨迹长度”而不是“病人状态演化”。

### 4.1.3 增加 missing-aware 状态记忆与质量分数

新版为 `tumor_status / ecog / karnofsky` 增加了 carry-forward 和记忆时间，并构造 `trajectory_quality_score`，用于区分：

- 高质量数据：走主线策略。
- 低质量数据：走更保守的 fallback 策略。

定义：

`trajectory_quality_score = mean_trajectory_length * state_quality_mean`

这里：

- `mean_trajectory_length` 越大，说明可学习的动态信息越多。
- `state_quality_mean` 越高，说明观测新鲜度和缺失情况越好。

当 `trajectory_quality_score < tau_q` 时，系统自动切换到退化数据模式，默认阈值 `tau_q = 1.0`。

## 4.2 动力学模型（Transformer）修复与增强

动力学模型学的是：

`hat{s}_{t+1} = f_theta(s_{1:t}, a_{1:t})`

含义：用历史状态与历史动作预测下一时刻状态。

新版的主要调整：

1. 修复了上游状态构造 bug，这是最关键的提升来源。
2. 在 `models.py` 中增加了可选结构开关：
   - `transformer_norm_first`
   - `embedding_dropout`
3. 训练默认更稳：
   - `lr = 5e-4`
   - `weight_decay = 1e-3`
   - 先按时间排序，再在 epoch 间 shuffle sequence

这些改动的数学含义：

- `norm_first`：把 LayerNorm 前置，改善深层 Transformer 的梯度稳定性。
- `embedding_dropout`：在输入嵌入层做正则，降低对局部 token/状态偶然模式的过拟合。
- `weight_decay`：相当于对参数范数加惩罚，抑制过大权重。

严格 old/new 多种子结果：

- one-step MSE：`0.011236 -> 0.005306`
- one-step MAE：`0.026319 -> 0.016023`
- rollout MSE@5：`0.010088 -> 0.004973`
- rollout MSE@10：`0.009725 -> 0.004989`

这说明新版数字孪生的状态演化预测已经明显优于旧版。

## 4.3 奖励、Q 学习与 Offline Policy 训练修复

### 4.3.1 基本 Bellman 目标

新版和旧版的 Q 学习都围绕 Bellman 目标，但新版把训练、验证、评估的口径对齐了：

`y_t = tilde{r}_t + gamma (1 - d_t) Q_{bar{psi}}( s_{t+1}, argmax_a Q_psi(s_{t+1}, a) )`

其中：

- `tilde{r}_t`：归一化后的奖励。
- `Q_{bar{psi}}`：target network。
- `argmax_a Q_psi(...)`：Double-DQN 结构中由当前网络选动作。
- `d_t`：done mask。

Bellman 拟合损失：

`L_B = smooth_l1( Q_psi(s_t, a_t), y_t )`

这里 `smooth_l1` 比单纯 MSE 更稳，对异常误差更不敏感。

### 4.3.2 保守 Q 学习（CQL）

Conservative Q-Learning 的目标是防止离线数据分布之外的动作被高估：

`L_CQL = E_s [ log sum_a exp(Q_psi(s,a)) - Q_psi(s,a_t) ]`

解释：

- `log sum exp` 会把所有动作的 Q 值整体抬起来看。
- 再减去数据行为动作 `a_t` 的 Q 值。
- 若模型对未见动作过于自信，这一项就会变大，从而被惩罚。

### 4.3.3 行为克隆暖启动

新版保留了 BC warmup，但系数设计更保守、更适配真实数据：

`L_BC = CE( pi_psi(.|s_t), a_t )`

总损失可以写成：

`L = L_B + alpha L_CQL + beta_t L_BC`

其中：

- `alpha` 对应 `cql_weight`
- `beta_t` 在 warmup 阶段较大，之后衰减到较小尾项

新版默认：

- `cql_weight = 0.25`
- `bc_warmup = 2500`
- `bc_start_coef = 0.5`
- `bc_tail_coef = 0.2`
- `lr = 8e-5`

算法含义：

- 相比旧版更强的 BC 暖启动，能减少小数据下策略过快偏离行为分布。
- 更高的 CQL 权重让 policy 在真实医疗数据下更保守，避免高熵漂移。

### 4.3.4 决策视图与动力学视图解耦

新版把 dynamics state 和 decision state 做了部分解耦。原因是：

- 某些静态变量对动力学预测有帮助。
- 但它们对 reward/Q 学习会引入 cohort-level confounding。

所以新版对 decision model 引入 `state_override`：

- 对 `age / gender / grade / stage` 这类静态维度，可替换为训练集均值。
- dynamics 仍保留完整状态。

这不是“删特征”，而是让 `Q_psi(s,a)` 不过度依赖一些会诱导捷径学习的静态变量。

### 4.3.5 训练协议修复

新版做了这些关键修复：

1. train/val 改成 trajectory-level split。
2. validation Bellman target 与训练一致。
3. policy 选模时取消错误 safety 截断。
4. best checkpoint 依据 validation policy return，而不是只看 Bellman loss。
5. dataloader worker 改为可控，默认 `0`，避免受限环境里 multiprocessing 直接炸掉实验。

### 4.3.6 Offline 稳定性结果

多种子稳定性结果：

- 旧版 baseline：
  - mean return = `9.84`
  - return std = `26.22`
- 新版稳定默认：
  - mean return = `21.14`
  - return std = `12.25`
  - sharpe-like = `4.52`
  - safety = `1.0`

这说明新版不只是“平均值更高”，而是**方差显著下降**，更接近真实部署需要的稳定性。

## 4.4 退化数据的质量感知 fallback

真实部署不可能只有干净数据，所以新版加入了自动数据质量切换。

### 4.4.1 自动模式选择

定义：

- 若 `trajectory_quality_score >= tau_q`，选 `stable_default`
- 若 `trajectory_quality_score < tau_q`，选 `quality_aware_fallback`

这一步解决的是：不同数据质量 regime 不应该用同一条训练和决策路径。

### 4.4.2 质量感知 reward/Q 编码

低质量 fallback 不再只用原始状态，而是给 reward/Q encoder 增加质量门控特征。后续又升级为 `triple` 模式：

`x_RQ = [ s, q ⊙ s, (1-q) ⊙ s ]`

其中：

- `q`：逐维质量/新鲜度权重。
- `q ⊙ s`：高质量部分。
- `(1-q) ⊙ s`：低质量残差部分。

算法含义：

- 第一块保留原始信息。
- 第二块强调“可以信”的维度。
- 第三块显式告诉模型“哪些维度是不可靠的”。

### 4.4.3 低质量动作优势惩罚

在低质量状态下，新版增加动作优势惩罚，抑制模型在脏输入上产生过度自信的动作分离：

`A(s,a) = Q(s,a) - (1/|A|) sum_{a'} Q(s,a')`

新增惩罚项可写成：

`L_lowq = lambda E[ w(q_t) ||A(s_t, .)||_2^2 ]`

其中：

- `w(q_t)` 在低质量状态更大。
- `lambda` 对应 `RL_LOW_QUALITY_ADV_PENALTY`。

算法含义：

- 当输入质量差时，不鼓励网络把某个动作抬得特别高。
- 本质上是一种 uncertainty-aware conservatism。

### 4.4.4 退化数据结果

自动质量切换后的 offline 结果：

- clean auto-mode：
  - mean return = `20.76`
  - std = `17.38`
- clean stable default：
  - mean return = `19.38`
  - std = `14.46`

clean 上 auto-mode 没有明显坏，但也没有压倒性更强，因此高质量数据仍以 `stable_default` 为主。

degraded 上差异更关键：

- 旧 degraded baseline：
  - mean return = `2.88`
  - std = `31.14`
  - worst seed = `-32.15`
- auto degraded：
  - mean return = `0.91`
  - std = `4.96`
  - worst seed = `-6.11`

这代表：

- 均值没有继续提高。
- 但**最坏情况和方差被大幅压缩**。

对于部署，这个变化非常重要，因为医疗推荐系统不能只看平均最好，要看最坏情形是否失控。

## 4.5 Online BCQ、CQL 与系统耦合修复

### 4.5.1 BCQ 与 CQL 的含义

- `CQL`：通过保守惩罚抑制离线分布外高估，适合纯离线保守学习。
- `BCQ`：通过行为约束和候选动作过滤，让在线小步更新更贴近历史行为支持集。

在 online adaptation 阶段，BCQ 更适合做“在已有离线策略附近的小步保守更新”。

### 4.5.2 路由和工件路径修复

新版修掉了这些关键问题：

1. `ours/bcq` 不再误映射成 `cql`。
2. `best_bcq_policy.d3` 不再只认 `./output/models`，而是优先：
   - 显式 `--model-dir`
   - `RL_MODEL_DIR`
   - `RL_BCQ_POLICY_PATH`
3. 当 BCQ 不可用时会显式说明 fallback 原因，而不是静默退化。

### 4.5.3 BCQ action-space 重建修复

新版 BCQOnlineTrainer 会锁定期望 `action_size`，避免在线 batch 较窄时只根据局部出现动作重建策略实现。否则算法会在一次窄 batch 更新后忘掉完整动作空间。

### 4.5.4 主 online 协议改为低耦合 `5x5 sync benchmark`

新版把算法收益测试和 front-end 测试分开：

- 主算法 benchmark：`5 episodes x horizon 5` 的同步 update 协议。
- front-end benchmark：单独测 query、throughput、线程稳定性。

这样做的算法意义是：

- 主 benchmark 只回答“在相同在线新数据和相同更新次数下，BCQ 与 CQL 谁更好”。
- 不让 active learning、线程、expert simulator 的波动污染算法结论。

### 4.5.5 clean real online 结果

在真实 clean 数据、`5x5` 主协议下：

- BCQ mean post-return = `2.9121`
- CQL mean post-return = `2.6238`
- safety = `1.0`

说明：clean 数据上，`RL0305` 的 online 主链路已经成立，且 BCQ 略优于 CQL。

### 4.5.6 degraded online 结果

在 degraded 数据的 auto-quality online fast 验证中：

- BCQ：
  - mean post-return = `-1.0078`
  - std = `0.6020`
  - worst = `-1.8573`
- CQL：
  - mean post-return = `-0.6404`
  - std = `1.3185`
  - worst = `-2.4839`

解释：

- degraded 条件下，BCQ 的均值还没有赢过 CQL。
- 但 BCQ 的方差更小、worst-case 更好。
- 这说明新版 degraded fallback 已经把系统推到了“更稳但还不够强”的阶段。

## 4.6 Front-end 与 Active Learning 修复

新版 front-end 主要修了三类问题：

1. benchmark 结束过早，后台 update 还没来得及计入。
2. `query_batch` 以前只是记成已查询，但整批样本没有真正逐条送到 expert。
3. `hybrid` sampler 的 batch flush 规则太僵硬，导致高价值候选滞留。

对应修复：

- 增加 update-wait 窗口。
- `query_batch` 改为整批逐条送标。
- `ExpertSimulator` 支持多 worker。
- `hybrid sampler` 支持 `partial flush` 和 `tail flush`。

算法含义：

- 这不是“刷系统吞吐”，而是提高有效 label 覆盖率，让 online update 真正拿到足够标注样本。

修复后，front-end 的 labeled buffer 和真实 updates 都明显上升，不再出现“统计上 query 很多，实际上没送多少标”的伪进展。

## 5. 现在的效果到底有多好

## 5.1 已经明显变好的部分

### 5.1.1 数字孪生动力学预测

这是新版最确定、证据最硬的一部分：

- `RL0305` 的 dynamics 明显优于 `RL1128`
- one-step 和 rollout 误差都大幅下降
- 说明 digital twin 的状态演化底座已经被真正修好

### 5.1.2 Offline 稳定性

- 旧版 old baseline：`mean 9.84 / std 26.22`
- 新版 stable default：`mean 21.14 / std 12.25`

新版不只是更高，而且更稳。

### 5.1.3 clean real online

- BCQ 比 CQL 略优
- safety 稳定为 `1.0`
- 说明在线主路径已经能工作，不再是假性 fallback 或路径错配

## 5.2 还没有完全解决的部分

### 5.2.1 degraded 数据上的平均收益

虽然新版把最坏情况从 `-32.15` 拉到了 `-6.11`，这是很大的安全改进；但 degraded offline 的平均收益没有同步提高，degraded online 仍为负值。

这意味着：

- 旧版的问题是“会失控”。
- 新版目前是“不会那么失控了，但还不够好”。

### 5.2.2 任意 treatment 数据泛化

当前系统还没有资格宣称：

“任意 treatment 相关数据进来，都能自动维护出高质量 digital twin，并给出稳定优秀推荐。”

原因不是代码还没通，而是统计学习边界还没跨过去。当前系统仍依赖以下前提：

1. 新数据和当前 ovarian treatment 数据在语义上足够接近。
2. 状态变量可以映射到同一套 state/action/reward 定义。
3. 轨迹长度、缺失模式、动作支持集不能偏离太多。

如果新数据集的治疗动作空间、结局定义、时间粒度都变了，当前系统不会自动泛化成一个完全可靠的新 digital twin。

## 6. 是否可以稳定部署

### 6.1 可以说“接近可部署”的部分

对于**同域、同语义、质量较好的真实治疗时序数据**，当前系统已经具备这些条件：

- 动力学模型稳定
- offline policy 相比旧版稳定很多
- online BCQ 路由正确
- front-end 和 active learning 的耦合问题基本收住
- clean real 数据上 offline + online 都已有正向结果

所以对这类数据，可以说：

**接近可部署，但仍应配 acceptance gate 和人工审查。**

### 6.2 还不能说“稳定部署”的部分

对于**退化真实数据**和**未知 treatment 数据分布**，当前系统还不能直接宣称稳定部署。原因是：

1. degraded offline/online 的均值收益还不够强。
2. auto-quality fallback 主要提升的是鲁棒性，不是平均策略质量。
3. 任意新治疗数据可能存在 schema、ontology、时间粒度、reward 可观测性等根本不一致。

因此更准确的部署判断是：

- `RL1128`：不建议部署。
- `RL0305`：
  - 对当前目标域的 clean/high-quality 数据，可进入受控部署或 shadow deployment。
  - 对 degraded/unknown-quality 数据，只能进入 guarded deployment，必须启用质量检测、fallback 和人工审查。
  - 对任意新 treatment 数据集，暂时不能直接部署，必须先做数据接入校验和小规模离线验证。

## 7. 建议的部署门槛

如果要真部署，至少建议加三层门槛：

### 7.1 数据接入门槛

新数据进入系统前先检查：

- 轨迹平均长度是否足够
- 缺失率是否超过阈值
- 动作空间是否落在已支持治疗集合内
- 奖励/结局是否可观测
- `trajectory_quality_score` 是否达到上线要求

### 7.2 不确定性/回退门槛

当满足以下任一条件时，不给激进推荐，只允许保守策略或人工接管：

- 质量分数低于阈值
- Q 动作间距过大但输入质量差
- 新数据显著偏离训练域

### 7.3 新数据集冷启动验证

任意新 treatment 数据集都应先过：

1. 小规模 offline benchmark
2. auto-quality profiling
3. 必要时 degraded fallback 测试
4. 再决定能否进入 online 更新

## 8. 最终判断

一句话总结：

- `RL1128` 的问题主要是“数据构造错、训练评估口径不一致、online 路由和 front-end 耦合重”，导致系统看起来在跑，实则很多结论不可靠。
- `RL0305` 已经把这些主问题大部分修掉了，clean real 数据上效果明显更好，也更稳定。
- 但当前最稳妥的结论是：

**`RL0305` 已经具备“在当前目标域高质量数据上受控部署”的基础，但还没有达到“任意 treatment 数据都能稳健泛化并自动给出高质量推荐”的程度。**
